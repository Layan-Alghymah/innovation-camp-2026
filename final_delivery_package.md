# حزمة التسليم النهائية لموقع **معسكر الابتكار الرقمي والمفتوح (2) - 2026**

## 1. نظرة عامة على حزمة التسليم

تتضمن هذه الحزمة الشفرة الكاملة لموقع React الجاهز للنشر، مع دليل تشغيل وصيانة يوضح كيفية بناء المشروع وتشغيله ونشره، وكيفية تحديث العناصر الأكثر حساسية للتغيير التشغيلي، وهي **صورة الـ Hero**، و**تاريخ العدّاد التنازلي**، و**رابط التسجيل الخارجي**، و**الألوان الأساسية**، و**المحتوى النصي للأقسام**. كما تتضمن الحزمة إرشادات صريحة للحفاظ على **سلوك RTL**، و**عمل شريط تقدم التمرير**، بالإضافة إلى **قائمة تحقق تنفيذية** يمكن استخدامها قبل أي إطلاق جديد أو بعد أي تعديل محتوى.

| عنصر التسليم | الوصف | المسار أو الصيغة |
|---|---|---|
| الشفرة الكاملة | نسخة المشروع القابلة للتشغيل محليًا أو إعادة رفعها | ملف ZIP مرفق |
| دليل التسليم | تعليمات النشر والصيانة والتحديث والتحقق | `final_delivery_package.md` |
| مرجع المحتوى | المصدر النصي الموحد للمحتوى العربي المعتمد | `hackathon_info.md` |
| نسخة المشروع السحابية | لقطة قابلة للفتح من بيئة المشروع الحالية | مرفق مستقل |

## 2. بنية المشروع الأساسية

يعتمد المشروع على **React + Vite** في الواجهة الأمامية، مع خادم Express بسيط لتقديم ملفات البناء في وضع الإنتاج. صفحة الموقع الأساسية موجودة حاليًا في ملف واحد رئيسي لتسهيل التسليم والصيانة السريعة، بينما تُحفظ الأصول النصية المرجعية في ملف منفصل داخل المشروع.

| المسار | الغرض |
|---|---|
| `client/src/pages/Home.tsx` | الصفحة الرئيسية الكاملة، وتشمل الأقسام والمحتوى والتفاعلات والثوابت القابلة للتحديث |
| `client/src/index.css` | نظام الألوان، الأنماط العامة، الحركة، معالجة RTL البصرية، وضبط المسافات والاستجابة |
| `client/index.html` | إعداد اللغة العربية والاتجاه RTL والخطوط والميتا الأساسية |
| `hackathon_info.md` | ملف المحتوى المرجعي الموحّد للنصوص والمعلومات المعتمدة |
| `package.json` | أوامر التشغيل والبناء والمعاينة |
| `server/index.ts` | تشغيل نسخة الإنتاج المبنية بعد تنفيذ أمر البناء |

## 3. متطلبات التشغيل المحلي

لتشغيل المشروع محليًا، يكفي وجود **Node.js** و**pnpm**. بعد فك ضغط الشفرة، انتقل إلى مجلد المشروع وثبّت الاعتمادات، ثم شغّل بيئة التطوير. لا يحتاج الموقع إلى قاعدة بيانات أو خدمات خلفية إضافية لأن التسجيل يتم خارجيًا عبر Google Form.

| الخطوة | الأمر |
|---|---|
| تثبيت الاعتمادات | `pnpm install` |
| تشغيل بيئة التطوير | `pnpm dev` |
| التحقق من الأنواع | `pnpm check` |
| إنشاء نسخة الإنتاج | `pnpm build` |
| تشغيل نسخة الإنتاج | `pnpm start` |
| معاينة البناء محليًا | `pnpm preview` |

## 4. تعليمات النشر

يمكن نشر المشروع بطريقتين عمليتين. الأولى من داخل بيئة المشروع الحالية عبر النشر المدمج بعد حفظ نقطة استعادة، والثانية خارج البيئة من خلال تشغيل البناء ورفع الناتج إلى منصة استضافة تدعم تطبيقات Vite/Node. بما أن المشروع حاليًا مبني كواجهة React ثابتة مع خادم تقديم ملفات بسيط، فإن النشر لا يتطلب أي تهيئة معقدة طالما أن أوامر البناء والتشغيل أعلاه متاحة.

### 4.1 النشر من داخل بيئة المشروع الحالية

افتح المشروع من النسخة السحابية المرفقة، ثم استخدم زر **Publish** بعد اختيار النسخة المناسبة من سجل الإصدارات. هذا هو المسار الأكثر مباشرة إذا كنت تريد إبقاء المشروع داخل نفس بيئة الإدارة.

### 4.2 النشر من خارج البيئة

بعد فك ضغط المشروع محليًا، نفّذ `pnpm install` ثم `pnpm build`. بعد ذلك شغّل `pnpm start` لاختبار نسخة الإنتاج. إذا كانت منصة الاستضافة تتطلب أمر بناء وتشغيل منفصلين، فاستخدم:

| الإعداد | القيمة |
|---|---|
| Build Command | `pnpm build` |
| Start Command | `pnpm start` |
| Dev Command | `pnpm dev` |
| Node Version | يوصى بإصدار حديث متوافق مع React 19 وVite 7 |

## 5. التهيئة والعناصر القابلة للتحديث

أهم ما يجب معرفته عند صيانة هذا المشروع أن معظم الإعدادات التشغيلية الحساسة موجودة مباشرة داخل ملف `client/src/pages/Home.tsx`، بينما نظام الهوية البصرية متمركز في `client/src/index.css`. لذلك فإن أي تحديثات تشغيلية مستقبلية ستكون واضحة إذا التزم فريق الصيانة بهذين الملفين بوصفهما المصدرين الرئيسيين للسلوك والمظهر.

### 5.1 تحديث صورة الـ Hero

صورة الخلفية الرئيسية محفوظة كثابت داخل الملف `client/src/pages/Home.tsx` تحت الاسم **`HERO_IMAGE`**. لتغييرها، استبدل الرابط الحالي برابط الصورة الجديدة فقط، مع الحفاظ على الطبقة الشفافة أعلى الصورة كما هي، لأن هذه الطبقة هي التي تضمن وضوح النص والعدّاد فوق الخلفية عبر مختلف الشاشات.

| العنصر | الموضع | ما الذي يجب تغييره؟ |
|---|---|---|
| صورة الـ Hero | `const HERO_IMAGE = ...` | استبدال رابط الصورة فقط |
| طبقة الوضوح | عنصر overlay داخل `HeroSection` | لا تُزلها، ويمكن فقط ضبط شفافيتها عند الضرورة |

### 5.2 تحديث تاريخ العدّاد التنازلي

العدّاد يعمل من خلال الاستدعاء التالي داخل `HeroSection`:

```tsx
const countdown = useCountdown("2026-05-12T18:00:00+03:00");
```

عند الحاجة إلى تغيير تاريخ أو وقت العدّاد، استبدل القيمة النصية فقط بصيغة تاريخ ووقت واضحة، ويفضل الإبقاء على المنطقة الزمنية ضمن السلسلة النصية لتجنب أي اختلاف بين البيئات.

| العنصر | الموضع | مثال تحديث |
|---|---|---|
| تاريخ العدّاد | `useCountdown("...")` | `useCountdown("2026-06-01T09:00:00+03:00")` |

### 5.3 تحديث رابط التسجيل الخارجي

رابط التسجيل الخارجي محفوظ كثابت باسم **`GOOGLE_FORM_URL`** في أعلى ملف `client/src/pages/Home.tsx`. جميع أزرار التسجيل تعتمد عليه، لذلك فإن تغييره مرة واحدة ينعكس تلقائيًا على كامل الموقع.

```tsx
const GOOGLE_FORM_URL = "https://...";
```

يجب عدم استبدال هذا المنطق باستمارة داخلية. كما يجب الحفاظ على فتح الرابط في **تبويب جديد** عبر خصائص الرابط المستخدمة في المكوّنات الحالية.

| العنصر | الموضع | الملاحظة |
|---|---|---|
| رابط التسجيل | `const GOOGLE_FORM_URL = ...` | غيّر الرابط مرة واحدة فقط |
| سلوك الفتح | `target="_blank"` و`rel="noopener noreferrer"` | يجب الإبقاء عليهما كما هما |

### 5.4 تحديث الألوان والثيم

نظام الألوان معرّف في `client/src/index.css` داخل `:root`. إذا كان المطلوب تعديل الدرجة اللونية دون كسر التصميم، فيجب تعديل المتغيرات التالية أولًا: **`--brand-blue`**، **`--brand-gold`**، **`--brand-orange`**، **`--page-bg`**، و**`--text-soft`**. يفضل عدم استبدال القيم مباشرة داخل المكوّنات قبل تحديث المتغيرات الأساسية.

| المتغير | الاستخدام |
|---|---|
| `--brand-blue` | اللون الأساسي للعناوين والهرمية البصرية |
| `--brand-gold` | لون التحويلات والعناصر البارزة |
| `--brand-orange` | امتداد تدرجات الدعوة إلى الإجراء |
| `--page-bg` | الخلفية العامة الفاتحة للموقع |
| `--text-soft` | النصوص الثانوية والوصفية |

### 5.5 تحديث المحتوى النصي للأقسام

معظم المحتوى النصي موجود داخل مصفوفات وثوابت في `client/src/pages/Home.tsx`، بينما مرجع المحتوى النصي الرسمي موجود في `hackathon_info.md`. أفضل ممارسة للصيانة هي تحديث `hackathon_info.md` أولًا، ثم عكس التغيير في الثوابت المقابلة في الصفحة.

| القسم | مكان التعديل الرئيسي |
|---|---|
| الأهداف | `const objectives` |
| الأنشطة | `const activities` |
| المسارات والتحديات | `const tracks` |
| الجدول الزمني | `const timeline` |
| شروط المشاركة | `const requirements` |
| معايير التقييم | `const criteria` |
| التقنيات الداعمة | `const technologies` |
| الفئات المستهدفة | `const audience` |
| الجوائز | `const awards` |
| الأسئلة الشائعة | `const faqItems` |
| المرجع النصي المعتمد | `hackathon_info.md` |

## 6. الحفاظ على RTL بصورة صحيحة

يدعم المشروع الاتجاه من اليمين إلى اليسار على مستوى الوثيقة وعلى مستوى التطبيق. أول طبقة يجب المحافظة عليها هي إعداد `dir="rtl"` و`lang="ar"` داخل `client/index.html`. الطبقة الثانية هي إبقاء الغلاف الجذري للصفحة يحمل `dir="rtl"` داخل `Home.tsx`. أي تعديل مستقبلي على المكوّنات يجب أن يحافظ على هذه البنية، وألا يعيد إدخال تخطيطات مبنية على افتراض LTR دون ضبط صريح.

| موضع الإعداد | المطلوب الحفاظ عليه |
|---|---|
| `client/index.html` | `<html lang="ar" dir="rtl">` |
| `Home.tsx` | `<div dir="rtl" ...>` في الجذر |
| النصوص الطويلة | الحفاظ على التفاف النص وعدم استخدام حاويات ضيقة جدًا |
| الأزرار والتنقل | اختبار ترتيب العناصر بصريًا بعد أي تعديل هيكلي |

إذا تمت إضافة مكوّنات جديدة مستقبلًا، فيجب اختبار **المحاذاة**، و**ترتيب الأيقونات بالنسبة للنص**، و**سلوك القوائم الأفقية**، خاصة عند المقاسات المتوسطة والصغيرة.

## 7. الحفاظ على سلوك شريط تقدم التمرير

يعتمد شريط التقدم العلوي على الدالة `useScrollProgress()` داخل `Home.tsx`، ويحسب النسبة بناءً على `window.scrollY` مقارنة بطول الصفحة الفعلي. للحفاظ على هذا السلوك، لا ينبغي تحويل الصفحة إلى حاوية تمرير داخلية معزولة أو جعل المحتوى يُمرَّر داخل عنصر آخر بدل الصفحة نفسها، لأن ذلك سيجعل الحساب الحالي غير دقيق ما لم يتم تعديل المنطق.

| العنصر | الموضع | ملاحظة الصيانة |
|---|---|---|
| منطق الحساب | `useScrollProgress()` | اتركه مرتبطًا بتمرير الصفحة الرئيسية |
| شريط التقدم | `ScrollProgressBar()` | يفضّل إبقاؤه مثبتًا أعلى الصفحة |
| السلوك الصحيح | تمرير الصفحة نفسها | لا تنقل التمرير إلى حاوية داخلية إلا مع تحديث المنطق |

## 8. ملاحظات تشغيلية مهمة

يعتمد الموقع على روابط صور خارجية جاهزة بدل تخزين الأصول الثقيلة داخل مجلد الواجهة، وهذا مناسب للنشر الحالي. عند استبدال أي صورة رئيسية مستقبلًا، استخدم رابطًا ثابتًا وموثوقًا للصورة النهائية، ثم اختبر وضوح النص فوقها، خصوصًا في الـ Hero وفي قسم الدعوة إلى التسجيل. كما أن الموقع لا يرسل بيانات تسجيل ولا يستقبلها؛ لذلك فإن أي تغيير في سياسة التسجيل يجب أن يبدأ من تحديث الرابط الخارجي فقط ما لم يتغير نطاق المشروع بالكامل.

## 9. قائمة التحقق قبل النشر أو بعد أي تعديل

يُنصح باستخدام القائمة التالية في كل مرة يتم فيها تحديث محتوى أو صورة أو تاريخ أو رابط تشغيل، حتى يبقى المشروع سهل الصيانة وخاليًا من الالتباس.

| بند التحقق | ما الذي يجب التأكد منه؟ | الحالة |
|---|---|---|
| عرض الـ Hero | الخلفية واضحة، والطبقة الشفافة تحافظ على قراءة العنوان والنص | ☐ |
| العدّاد التنازلي | يظهر أرقامًا صحيحة، ويتحدث كل ثانية، ويظل في المركز عبر الشاشات | ☐ |
| التنقل العلوي | الروابط تنتقل إلى الأقسام الصحيحة، ويظهر تمييز القسم النشط | ☐ |
| رابط التسجيل الخارجي | جميع أزرار التسجيل تفتح Google Form في تبويب جديد | ☐ |
| شريط التقدم | يتحرك بسلاسة ويعكس طول الصفحة الفعلي | ☐ |
| RTL | المحاذاة والقراءة واتجاه العناصر صحيحة في كل الأقسام | ☐ |
| النصوص الطويلة | لا يوجد كسر تخطيط في البطاقات أو الخط الزمني أو الأسئلة الشائعة | ☐ |
| الاستجابة | التجربة مستقرة على الجوال واللوحي وسطح المكتب | ☐ |
| البناء | أمر `pnpm build` يعمل دون أخطاء | ☐ |

## 10. خطوات صيانة سريعة بدون غموض

إذا كان المطلوب تعديل تشغيلي سريع، فاستخدم هذا التسلسل المختصر: حدّث المرجع النصي في `hackathon_info.md` إذا كان التغيير محتوى، ثم عدّل الثابت أو المصفوفة المقابلة في `Home.tsx`. بعد ذلك راجع `index.css` فقط إذا كان التعديل بصريًا. ثم شغّل `pnpm build` للتحقق، وافتح المعاينة للتأكد من وضوح الواجهة، وصحة RTL، وعمل التسجيل الخارجي، ثم انشر النسخة المحدثة.

بهذا الترتيب، يبقى المشروع **سهل النشر والصيانة**، ويمكن التعامل معه بسرعة دون الحاجة إلى إعادة تصميم أو إدخال تغييرات بنيوية غير لازمة.
