# معمارية النظام وبنية الواجهة لموقع **معسكر الابتكار الرقمي والمفتوح (2) - 2026**

## 1) نطاق الوثيقة

تحدد هذه الوثيقة **المعمارية الأمامية الكاملة** لموقع ويب عربي احترافي باتجاه **RTL**، مخصص لفعالية **معسكر الابتكار الرقمي والمفتوح (2) - 2026**، بحيث تكون جاهزة للتنفيذ باستخدام **React** من دون الدخول في أي تفاصيل خاصة بالخلفية البرمجية أو قواعد البيانات أو الخدمات الخلفية. وتركز الوثيقة على أربع طبقات مترابطة: **هيكل الصفحة، تسلسل الأقسام، بنية المكوّنات، ونموذج التفاعل**، مع تعريف واضح لنظام الثيم، وسلوك RTL، وآلية الربط الخارجي بنموذج التسجيل.

تم اعتماد التوجه التصميمي المختار من ملف العصف البصري بوصفه **"الضوء المؤسسي المتقدّم"**، لأنه يحقق التوازن الأفضل بين الرصانة المؤسسية، والهوية التقنية، والطابع المستقبلي المرتبط بالطاقة والبحث والابتكار، من دون أن يتحول الموقع إلى لوحة تحكم جافة أو إلى صفحة تسويقية صاخبة.

## 2) الرؤية المعمارية العامة

الموقع يُبنى بوصفه **تجربة صفحة رئيسية طويلة Single-Page Landing Experience** ذات بنية معيارية، حيث تعمل الصفحة على عرض المحتوى عبر أقسام مرتبة ترتيبًا إلزاميًا، ويُدار التنقل داخليًا من خلال رأس ثابت وروابط انتقال ناعم إلى المقاطع. لا توجد طبقة حسابات أو نماذج إدخال داخلية أو إدارة حالة معقدة مرتبطة ببيانات مستخدم، لأن الهدف الأساسي هو **التعريف، الإقناع، والتنقّل الموجّه نحو تسجيل خارجي**.

لذلك فإن المعمارية المناسبة ليست معمارية تطبيق معقد متعدد الشاشات، بل **معمارية عرض تفاعلي modular content-driven frontend**، يكون فيها كل قسم وحدة مستقلة نسبيًا وقابلة لإعادة الاستخدام، مع وجود **طبقة Layout موحدة** تنسق الرأس، شريط التقدم، حاوية الصفحة، وانتقالات الأقسام.

| الطبقة | دورها في المعمارية |
|---|---|
| App Shell | يضبط الاتجاه RTL، الثيم الفاتح، التخطيط العام، وسلوك التنقل |
| Layout Layer | يحتضن الرأس الثابت، شريط التقدم، المحتوى الرئيسي، والتذييل |
| Section Layer | يمثل الأقسام الأساسية للموقع بالترتيب الإلزامي |
| Component Layer | يقدم الوحدات التفاعلية مثل العدّاد، بطاقات المسارات، الخط الزمني، والأسئلة الشائعة |
| Content Layer | يسحب النصوص والروابط والتواريخ من مصدر المحتوى الموحّد `hackathon_info.md` بعد تحويله إلى صيغة قابلة للاستهلاك في الواجهة |

## 3) بنية الصفحة العامة

يجب أن تحتوي الواجهة على **صفحة رئيسية واحدة** تمثل التجربة الكاملة، مع إمكانية لاحقة لإضافة صفحات مساندة مثل صفحة سياسة الخصوصية أو صفحة 404، لكن من دون أن تؤثر تلك الصفحات على المعمارية الأساسية للموقع. الصفحة الرئيسية يجب أن تبدأ بعناصر النظام الثابتة في الأعلى، ثم تمر بالأقسام بالمطلق وفق الترتيب المحدد في المواصفة الأصلية.

### الصفحة الأساسية المطلوبة

| المسار | الصفحة | الغرض |
|---|---|---|
| `/` | الصفحة الرئيسية | عرض كامل محتوى المعسكر، التفاعل، والتوجيه إلى التسجيل الخارجي |
| `/404` | صفحة غير موجود | صفحة افتراضية بسيطة خارج نطاق التصميم الرئيسي |

### تسلسل التجربة داخل الصفحة الرئيسية

| المستوى | الكيان | الوصف |
|---|---|---|
| 1 | Scroll Progress Bar | شريط تقدم ثابت في أعلى الشاشة يعكس موقع المستخدم داخل الصفحة |
| 2 | Sticky Header | ترويسة ثابتة مع تنقل RTL وإبراز القسم النشط |
| 3 | Main Content | الحاوية التي تضم الأقسام الأربعة عشر بالترتيب الرسمي |
| 4 | Footer | تذييل ختامي يحوي الروابط المختصرة والتنويه المؤسسي والرابط الخارجي |

## 4) التسلسل الهرمي للأقسام داخل الصفحة

يجب أن يُعرّف كل قسم بوصفه **مكوّن Section مستقل** له `id` خاص يُستخدم لأغراض الملاحة، وتتبع القسم النشط، والوصول المباشر، والتدرج الحركي أثناء التمرير. ولا يجوز تغيير ترتيب هذه الأقسام في الهيكل العام.

| الترتيب | المعرّف المقترح | القسم | الغرض المعماري |
|---|---|---|---|
| 1 | `hero` | Hero | إنشاء الانطباع الأول، إبراز العدّاد، وتأكيد الهوية والرابط الخارجي |
| 2 | `about` | About | تقديم المعسكر وسياقه وقيمته |
| 3 | `objectives` | Objectives | عرض الأهداف الستة كبنية معرفية واضحة |
| 4 | `activities` | Activities | شرح رحلة المعسكر وأنشطته الأربع |
| 5 | `tracks` | Tracks | عرض المسارات الأربع مع التحديات بأسلوب تفاعلي |
| 6 | `timeline` | Timeline | تمثيل المراحل العشر زمنيًا بشكل متدرج ومتحرك |
| 7 | `requirements` | Participation Requirements | توضيح شروط المشاركة والالتزامات |
| 8 | `criteria` | Evaluation Criteria | عرض معايير التحكيم الثمانية |
| 9 | `technologies` | Supporting Technologies | إبراز التقنيات الداعمة المقترحة |
| 10 | `audience` | Target Audience | تعريف الفئات المستهدفة |
| 11 | `awards` | Awards | عرض الجوائز أو فرص التمكين اللاحقة |
| 12 | `faq` | FAQ | الإجابة عن الأسئلة الشائعة عبر Accordion |
| 13 | `registration` | Registration CTA | إعادة توجيه واضحة إلى التسجيل الخارجي |
| 14 | `footer` | Footer | الإغلاق المؤسسي والتنقل المختصر |

## 5) معمارية التطبيق الأمامي في React

الموقع يجب أن يُنفذ وفق **معمارية مكونات واضحة ومنفصلة المسؤوليات**. والمقصود هنا ليس مجرد تقسيم ملفات، بل تقسيم منطقي يجعل كل جزء من النظام مسؤولًا عن دور واحد محدد: التخطيط، العرض، الحالة التفاعلية، أو البيانات. وبما أن المشروع لا يحتوي على منطق خلفي، فإن بنية React يجب أن تركز على **سهولة التوسعة، وقابلية الصيانة، ونظافة الاعتماديات**.

### البنية المقترحة للمجلدات

| المسار | الدور المعماري |
|---|---|
| `client/src/pages/` | الصفحات العليا مثل `HomePage` |
| `client/src/layout/` | مكونات التخطيط مثل `SiteShell`, `StickyHeader`, `Footer`, `SectionContainer` |
| `client/src/components/sections/` | أقسام الصفحة الرئيسية، وكل قسم ملف مستقل |
| `client/src/components/common/` | العناصر المشتركة مثل الأزرار الثانوية، شارات التواريخ، عناوين الأقسام، Decorative Grid |
| `client/src/components/interactive/` | العدّاد، شريط التقدم، Accordion، Timeline، Track Cards |
| `client/src/hooks/` | خطافات مثل `useActiveSection`, `useScrollProgress`, `useCountdown` |
| `client/src/lib/` | أدوات مثل `cn`, إعدادات الثيم، ثوابت الروابط، ومعالجات البيانات |
| `client/src/content/` | الملف المحوّل من `hackathon_info.md` إلى بنية بيانات قابلة للعرض |
| `client/src/types/` | تعريف أنواع البيانات الخاصة بالأقسام والعناصر التفاعلية |

## 8) نموذج التفاعل العام

التفاعل في هذا الموقع يجب أن يكون **مُوجّهًا لا مُبهِرًا**. الهدف ليس استعراض الحركة، بل تنظيم انتباه الزائر ومساعدته على عبور الصفحة بثقة. لذلك يجب أن تعتمد الواجهة على ثلاثة مستويات من التفاعل: **تفاعل الملاحة، تفاعل الاكتشاف داخل القسم، وتفاعل التحويل نحو التسجيل الخارجي**.

| مستوى التفاعل | الوصف | أمثلة |
|---|---|---|
| الملاحة | يساعد المستخدم على معرفة مكانه والتنقل بسهولة | الرأس الثابت، إبراز القسم النشط، التمرير الناعم |
| الاكتشاف | يكشف معلومات إضافية بطريقة تدريجية | توسع بطاقات المسارات، Accordion للأسئلة، إبراز عناصر الخط الزمني |
| التحويل | يدفع المستخدم لاتخاذ الإجراء الخارجي | أزرار التسجيل في Hero وRegistration CTA وFooter |

## 9) تصميم **Hero Component**

يجب أن يكون **Hero** مكوّنًا مركزيًا من الناحية المعمارية والبصرية، لأن كل التجربة تبدأ منه. بنية هذا المكوّن لا تقتصر على صورة خلفية ونص، بل تتكون من **طبقات واضحة** تضمن التحكم الكامل في القراءة، والهرمية، والاستجابة، والتفاعل.

### الطبقات المعمارية داخل Hero

| الطبقة | المكوّن | الوظيفة |
|---|---|---|
| 1 | `HeroBackgroundContainer` | حاوية ممتدة بعرض الشاشة وارتفاع بصري كبير، تحتضن الخلفية المرئية |
| 2 | `HeroBackgroundMedia` | العنصر البصري الرئيسي، سواء أكان صورة ثابتة أو طبقة مرئية مستقبلية |
| 3 | `HeroOverlay` | طبقة شبه شفافة لتحسين التباين وإضافة عمق بصري |
| 4 | `HeroContent` | حاوية النصوص والعناصر التفاعلية |
| 5 | `CountdownTimer` | العنصر البؤري المركزي والأكبر بصريًا |
| 6 | `ExternalRegistrationActions` | أزرار التسجيل الخارجية و/أو الاستكشاف |

### السلوك المطلوب داخل Hero

يجب أن يشغل القسم كامل عرض الشاشة، وأن يحقق حضورًا بصريًا قويًا على الشاشات الكبيرة من دون أن يفقد التوازن على الهاتف. يجب أن يتم **توسيط العدّاد زمنيًا وبصريًا** داخل الحاوية، وأن يكون أكبر عنصر بصري، مع الحفاظ على وجود العنوان والوصف في محور مركزي متماسك. ويجب ألا تُضاف عناصر ثانوية كثيرة في هذا القسم حتى لا تنافس العدّاد على الانتباه.

### مواصفة العدّاد التنازلي

| العنصر | التوجيه |
|---|---|
| الموقع | في مركز Hero بصريًا ومحوريًا |
| الحجم | كبير جدًا مقارنة بباقي العناصر |
| البنية | وحدات زمنية منفصلة: أيام، ساعات، دقائق، ثوانٍ |
| السلوك | تحديث زمني دوري على العميل فقط |
| الحالة بعد الانتهاء | التحول إلى رسالة مثل "بدأت الفعالية" أو الانتقال إلى أقرب محطة زمنية لاحقة |
| المصدر | التاريخ الهدف يُسحب من مصدر المحتوى المركزي |

### أزرار التسجيل داخل Hero

يجب أن تُستبدل أي صيغة تسجيل داخلي بمكوّن **ExternalRegistrationActions**، بحيث يحتوي على زر رئيسي وربما زر ثانوي. الزر الرئيسي يجب أن يفتح **Google Form في تبويب جديد**، عبر سلوك واضح وآمن. لا توجد أي حقول إدخال، ولا حالة "submitted" داخلية، ولا نماذج منبثقة.

| الزر | السلوك |
|---|---|
| الزر الرئيسي | `Open Google Form in new tab` |
| الزر الثانوي | انتقال داخلي إلى قسم المسارات أو الجدول الزمني |
| المنع الصريح | منع أي نموذج تسجيل محلي أو نافذة منبثقة لجمع البيانات |

## 10) تصميم **Sticky Header**

الرأس الثابت عنصر حاسم في المعمارية لأنه يربط كامل التجربة الطويلة بمركز تنقل واضح. يجب أن يكون **Sticky Header** مثبتًا أعلى الصفحة بعد شريط التقدم، وأن يحافظ على حضور خفيف وراقي من خلال خلفية شبه شفافة أو مموهة خفيفًا تسمح بالتثبيت من دون إحساس بالثقل.

### مكونات الرأس

| الجزء | المكوّن | الوظيفة |
|---|---|---|
| يمين الرأس | `BrandBlock` | شعار/اسم الفعالية أو الاسم المختصر |
| الوسط/اليسار وفق RTL | `PrimaryNav` | روابط الأقسام الأساسية |
| الحافة المقابلة | `HeaderActions` | زر التسجيل الخارجي وربما زر القائمة في الهاتف |

### سلوك الرأس في RTL

لأن الواجهة عربية RTL، يجب أن يبدأ التنظيم البصري من اليمين، وأن تكون المحاذاة والمنطق الحركي متوافقين مع ذلك. ترتيب عناصر القائمة يجب أن يحترم القراءة العربية، كما يجب أن يتحرك مؤشر القسم النشط أو خط التحديد بصريًا بطريقة متوافقة مع الاتجاه من اليمين إلى اليسار.

### إبراز القسم النشط

يجب أن يعتمد الرأس على **Active Section Highlighting** عبر مراقبة مواقع الأقسام أثناء التمرير. يوصى باستخدام **Intersection Observer** أو منطق معادل داخل Hook مثل `useActiveSection`. عند دخول قسم إلى منطقة التفعيل المرئية، تتغير حالة الرابط المناظر له داخل الرأس.

| حالة الرابط | السلوك البصري |
|---|---|
| افتراضي | لون هادئ وتباين منخفض نسبيًا |
| مرور Hover | ارتفاع طفيف أو توهج خفيف أو تغير لون accent |
| نشط Active | لون أساسي أقوى مع underline أو pill accent |
| هاتف محمول | القائمة تنتقل إلى Drawer أو Sheet مع الحفاظ على ترتيب RTL |

## 11) تصميم **Scroll Progress Bar**

يجب أن يكون شريط التقدم عنصرًا مستقلًا وثابتًا عند أعلى الشاشة، فوق الرأس أو ملاصقًا له، بحيث يعكس **النسبة المئوية الفعلية للتقدم الرأسي داخل الصفحة**. وهو عنصر ملاحي وإحساسي في آن واحد، لأنه يمنح المستخدم إدراكًا واضحًا للمسافة المتبقية.

### مواصفة شريط التقدم

| العنصر | المتطلب |
|---|---|
| التموضع | Fixed top |
| الارتفاع | رفيع وواضح، غير مبالغ فيه |
| اللون | accent ذهبي/برتقالي فوق قاعدة باهتة أو شفافة |
| القيمة | تتحدث بحسب scroll position |
| السلوك | انسيابي من دون تقطّع، ويفضّل smoothing بسيط |

المكوّن المناسب معماريًا هو `ScrollProgressBar`، ويعتمد على Hook مثل `useScrollProgress` لحساب `(scrollTop / scrollableHeight) * 100`. ويجب أن يُراعى ألا يتأثر الحساب بطول الرأس الثابت أو المسافات الداخلية غير المرئية.

## 12) تصميم **Track Cards**

قسم المسارات هو أحد أهم الأقسام لاكتشاف المحتوى، لذلك يجب أن تُصمم بطاقاته بوصفها **مكوّنات قابلة للتوسّع والاكتشاف التدريجي**. كل بطاقة تمثل مسارًا، وتعرض في حالتها الأساسية: اسم المسار، وصفًا موجزًا، ومؤشرًا بصريًا لطبيعته. ثم تكشف عند المرور أو التوسّع عن قائمة التحديات أو الأمثلة المرتبطة به.

### بنية `TrackCard`

| الجزء | الوظيفة |
|---|---|
| `TrackCardHeader` | اسم المسار، أيقونة أو رقم، وملخص سريع |
| `TrackCardBody` | وصف المسار وإطاره الابتكاري |
| `TrackChallengesPreview` | سطر أو سطران من التحديات في الحالة المختصرة |
| `TrackExpandRegion` | منطقة تتوسع لتعرض كامل التحديات |
| `TrackFooterMeta` | وسم تقني أو عبارة مثل "ذكاء اصطناعي" أو "استدامة" |

### نموذج التفاعل

على الشاشات الكبيرة، يمكن أن تعمل البطاقة بمنطق **hover + click** معًا. أما على الأجهزة اللمسية، فيجب أن يكون الاعتماد الأساسي على **click/tap expand**. لا يجوز أن تعتمد البطاقة على المرور فقط، لأن ذلك يكسر التجربة على الهاتف.

| الحالة | السلوك |
|---|---|
| Default | بطاقة مضغوطة بارتفاع ثابت ونصوص مختصرة |
| Hover Desktop | ارتفاع خفيف، ظل أعمق، كشف جزء إضافي من المحتوى |
| Expanded | ظهور قائمة التحديات كاملة مع انتقال سلس |
| Focus Keyboard | إطار واضح أو ring مطابق للثيم |

## 13) تصميم **Animated Timeline**

يجب أن يُقدَّم الخط الزمني بوصفه **مكوّنًا سرديًا متحركًا**، لا مجرد جدول جامد. الغرض منه أن يجعل الزائر يفهم كيف تمتد رحلة المعسكر من **19 أبريل 2026** إلى **12-13 مايو 2026** عبر عشر مراحل واضحة.

### نموذج العرض المقترح

يمكن تنفيذ الخط الزمني كـ **سكة زمنية رأسية على الهاتف وأفقية/هجينة على الشاشات الأوسع**، لكن بما أن RTL هو الأساس، فإن الاتجاه البصري في النسخة الأفقية يجب أن يبدأ من اليمين وينتقل يسارًا، بينما تبقى القراءة داخل كل بطاقة عربية ومحاذاة يمينية.

### مكونات الخط الزمني

| المكوّن | الوظيفة |
|---|---|
| `TimelineSectionHeader` | عنوان القسم وشرح موجز |
| `TimelineRail` | المسار البصري العام الذي يربط كل المراحل |
| `TimelineNode` | نقطة المرحلة أو مؤشرها |
| `TimelineCard` | بطاقة تحتوي التاريخ، العنوان، والغرض |
| `TimelineProgressGlow` | توهج أو مؤشر تقدم بصري يرتبط بالتمرير |

### التفاعل والحركة

| الحالة | السلوك |
|---|---|
| عند الظهور | كل مرحلة تدخل بتدرج محسوب لا دفعة واحدة |
| عند التمرير | المرحلة الحالية تزداد وضوحًا أو يزداد توهج مسارها |
| عند المرور | إبراز البطاقة وتفاصيلها |
| على الهاتف | البطاقات تتكدس عموديًا مع الاحتفاظ بخط ربط واضح |

## 14) تصميم **FAQ Accordion**

قسم الأسئلة الشائعة يجب أن يُبنى عبر **Accordion واضح ومريح بصريًا**، بحيث تكون كل وحدة سؤال/جواب مستقلة، وتحتفظ الواجهة بنظافتها من دون إغراق المستخدم بكل المحتوى دفعة واحدة. هذا القسم يخفف الحمل المعرفي في المراحل الأخيرة من الصفحة ويمنح المستخدم إجابات سريعة قبل التوجه إلى التسجيل الخارجي.

### مكونات Accordion

| الجزء | الوظيفة |
|---|---|
| `FaqAccordion` | الحاوية العامة |
| `FaqItem` | عنصر سؤال/جواب واحد |
| `FaqTrigger` | سطر السؤال القابل للضغط |
| `FaqContent` | الجواب الذي ينفتح تدريجيًا |
| `FaqIcon` | مؤشر بصري للدلالة على الفتح والإغلاق |

### سلوك Accordion

| الخاصية | التوجيه |
|---|---|
| نمط الفتح | Single أو Multiple حسب قرار التجربة، والأفضل Single للحفاظ على التركيز |
| الحركة | انسياب رأسي سلس دون قفز مفاجئ في التخطيط |
| الحالة النشطة | تمييز السؤال المفتوح بلون accent أو حد أو خلفية خفيفة |
| الوصول | دعم كامل للوحة المفاتيح وسمات ARIA |

## 15) انتقالات الأقسام **Section Transitions**

يجب أن تُعامل الانتقالات بين الأقسام بوصفها جزءًا من هوية التجربة، لكن من دون مبالغة. كل قسم ينبغي أن يظهر عند دخوله مجال الرؤية بانتقال **fade + translate** ناعم، مع فروقات طفيفة بين أنواع المحتوى. فمثلًا، الأرقام والمؤشرات قد تدخل بديناميكية أسرع من الفقرات الطويلة، بينما بطاقات المسارات يمكن أن تتتابع على شكل cascade RTL.

### التوجيه العام للانتقالات

| نوع العنصر | الانتقال المقترح |
|---|---|
| عناوين الأقسام | Fade up خفيف |
| الفقرات | Fade with small delay |
| البطاقات | Staggered reveal من اليمين إلى اليسار |
| الخط الزمني | Progressive draw / glow |
| العدّاد | حضور ثابت مع نبض خفيف غير مشتت |

## 16) نظام الثيم **Theming System**

الثيم يجب أن يُبنى كنظام تصميم متكامل، لا كمجرد ألوان متناثرة. المطلوب هو **Light Theme** بقاعدة بيضاء أو فاتحة جدًا، مع اعتماد لون أزرق مؤسسي داكن/قوي للثبات والثقة، ولون ذهبي أو عنبري للإبراز، مع أخضر خفيف جدًا فقط في سياقات الاستدامة أو مؤشرات تقنية محددة.

### طبقات الثيم الأساسية

| مستوى اللون | الغرض |
|---|---|
| Base Background | خلفية الصفحة والأقسام الرئيسية |
| Surface | البطاقات والرأس والتذييل والعناصر العائمة |
| Primary | الأزرار الرئيسية، الروابط النشطة، بعض العناوين أو الحدود المهمة |
| Accent | شريط التقدم، التحديدات، النقاط البارزة، مؤشرات الحالة |
| Neutral Text | النصوص الرئيسية والثانوية |
| Success/Green Subtle | إشارات الاستدامة أو النجاح بشكل محدود جدًا |

### المواصفة اللونية الوظيفية

| التوكن | الاستخدام |
|---|---|
| `--background` | أبيض أو أوف وايت بارد |
| `--foreground` | أزرق داكن/رمادي مائل للأزرق للقراءة الطويلة |
| `--primary` | أزرق مؤسسي قوي |
| `--primary-foreground` | أبيض نقي للعناصر النشطة |
| `--accent` | ذهبي/عنبر هادئ غير صارخ |
| `--muted` | أسطح فاتحة جدًا وخلفيات ثانوية |
| `--border` | حدود خافتة باردة |
| `--ring` | لون تركيز واضح ومتناغم مع primary/accent |

### المبادئ البصرية داخل الثيم

يجب أن تكون الحواف نظيفة لكن غير مبالغ في تدويرها، وأن تستخدم الظلال باعتدال لإضفاء العمق. وتُفضّل الأسطح شبه الزجاجية المحدودة فقط في الرأس وبعض البطاقات المميزة، مع تجنب المظهر الزجاجي الكثيف. ويجب أن تُستخدم المساحات البيضاء بوصفها أداة تنظيمية أساسية، لا مجرد فراغ.

## 17) سلوك RTL الكامل عبر المكوّنات

من الضروري ألا يقتصر RTL على جعل النصوص عربية فحسب، بل يجب أن يُطبّق على **البنية، الحركة، المحاذاة، الأولويات البصرية، وحالات التفاعل**. وهذا يعني أن المعمارية نفسها يجب أن تكون مدركة للاتجاه من البداية.

### قواعد RTL الإلزامية

| العنصر | سلوك RTL المطلوب |
|---|---|
| الصفحة | `dir="rtl"` على مستوى الجذر أو التطبيق |
| النصوص | محاذاة يمينية افتراضيًا إلا في الحالات الرقمية الخاصة |
| التنقل | يبدأ من اليمين، وتظهر الروابط بحسب المنطق العربي |
| الأيقونات الاتجاهية | تنعكس أو تُستبدل بنسخ تدعم RTL |
| الحركات المتتابعة | تدخل العناصر من اليمين إلى اليسار حيث يكون ذلك مناسبًا |
| البطاقات والجداول | تبدأ المعلومات الأساسية من يمين الحاوية |
| Drawer الهاتف | يفتح من الجهة المناسبة بصريًا مع الحفاظ على سهولة الوصول |

### ملاحظات خاصة بالأرقام والعدّاد

نظرًا لأن العدّاد عنصر رقمي بارز، يجب تحديد ما إذا كانت الأرقام ستعرض بصيغة عربية هندية أو لاتينية بحسب قرار الهوية التحريرية. لكن في كلتا الحالتين، يجب أن يظل **ترتيب وحدات الزمن منطقيًا وواضحًا للمستخدم العربي**، وأن تُضبط المحاذاة داخل كل وحدة زمنية بطريقة تمنع تشوه القراءة.

## 18) نموذج الحالة وإدارة السلوك في الواجهة

بما أن المشروع Frontend Only، فإن إدارة الحالة يجب أن تبقى خفيفة ومحلية قدر الإمكان. لا توجد حاجة إلى متجر حالة مركزي كبير إلا إذا توسعت التجربة مستقبلًا. في الوضع الحالي، تكفي **حالات محلية + Hooks مخصصة + ثوابت محتوى**.

| الحالة | المصدر المناسب |
|---|---|
| النسبة الحالية للتمرير | `useScrollProgress` |
| القسم النشط | `useActiveSection` |
| الزمن المتبقي في العدّاد | `useCountdown` |
| البطاقة الموسعة في المسارات | state محلي داخل `TracksSection` أو `TrackCardGroup` |
| العنصر المفتوح في FAQ | state محلي أو Accordion primitive |
| حالة قائمة الهاتف | state محلي داخل `StickyHeader` |

## 19) قابلية الوصول والوضوح

الموقع premium لا يعني موقعًا زخرفيًا فقط؛ بل يجب أن يكون سهل الاستخدام أيضًا. لذلك يجب أن تراعي المعمارية متطلبات الوصول على مستوى التباين، والتنقل بلوحة المفاتيح، وترتيب التركيز، وعناوين الأقسام، وتسمية الأزرار والروابط. كما يجب ألا تعتمد أي وظيفة أساسية على المرور فقط، خصوصًا في بطاقات المسارات وعناصر الأسئلة الشائعة.

| المجال | متطلب الوصول |
|---|---|
| التنقل | جميع الروابط والأزرار قابلة للوصول بلوحة المفاتيح |
| الرأس | يجب أن يتيح تخطي الملاحة أو الوصول السريع للمحتوى عند الحاجة |
| Accordion | سمات ARIA صحيحة وحالات مفتوح/مغلق واضحة |
| الألوان | تباين مناسب بين النص والخلفية في الثيم الفاتح |
| المؤثرات | احترام تفضيلات تقليل الحركة إذا لزم |

## 20) قيود التنفيذ المعماري

هذه الوثيقة تُلزم التنفيذ المستقبلي بمجموعة من القيود الواضحة، حتى لا يتحول المشروع أثناء التطوير إلى تطبيق مخالف للمواصفة الأساسية.

| القيد | التفسير |
|---|---|
| لا Backend | لا نماذج خادم، لا API داخلي، لا قواعد بيانات |
| لا تسجيل داخلي | جميع الدعوات إلى الإجراء تفتح Google Form خارجيًا فقط |
| لا نماذج إدخال رئيسية | الموقع تعريفي/تحفيزي وليس منصة إدارة مشاركات |
| لا كسر لترتيب الأقسام | الهيكل المعتمد إلزامي |
| لا خلط لغوي | الواجهة عربية RTL من الأصل |
| لا ثيم داكن كنسخة افتراضية | الثيم الأساسي Light فقط |
| لا صور محلية ثقيلة داخل المشروع | إن استُخدمت أصول مرئية لاحقًا في التنفيذ، تُدار وفق سياسة الأصول الخارجية المناسبة للمشروع |

## 21) الخلاصة التنفيذية الجاهزة لـ React

الموقع المطلوب جاهز للتنفيذ في React بوصفه **تجربة صفحة رئيسية عربية RTL ذات معمارية معيارية واضحة**. ويجب أن يتكون من **App Shell** يحتضن **Scroll Progress Bar** و**Sticky Header** و**Main Sections Renderer** و**Footer**، مع أقسام منفصلة تمثل: **Hero، About، Objectives، Activities، Tracks، Timeline، Participation Requirements، Evaluation Criteria، Supporting Technologies، Target Audience، Awards، FAQ، Registration CTA**. ويقود **Hero** التجربة عبر **حاوية خلفية ممتدة بعرض الصفحة مع Overlay**، و**عدّاد تنازلي كبير ومركزي**، وأزرار تسجيل خارجية فقط.

كما تعتمد التجربة على مكوّنات تفاعلية محددة بوضوح، تشمل: **بطاقات مسارات قابلة للتوسع والمرور**، و**خطًا زمنيًا متحركًا**، و**Accordion للأسئلة الشائعة**، و**انتقالات أقسام محسوبة**، مع **رأس ثابت يدعم الملاحة RTL وإبراز القسم النشط**، و**شريط تقدم علوي ثابت** يتحدث بحسب موضع التمرير. ويُبنى كل ذلك فوق **نظام ثيم فاتح** أساسه الأبيض مع **أزرق مؤسسي قوي** و**ذهبي/عنبر كلون إبراز**، مع التزام كامل بقواعد **RTL** على مستوى المحاذاة، الترتيب، التفاعل، والحركة.

بهذه الصورة، تصبح الوثيقة **مواصفة معمارية وواجهة كاملة** يمكن لفريق React أن يبدأ منها مباشرة، من دون حاجة إلى تفسير إضافي للهيكل أو منطق التفاعل أو قيود التسجيل أو أسس الثيم والتخطيط.
